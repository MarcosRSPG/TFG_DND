1. Introduccion

1.1 Contexto del proyecto

Grimledger es un proyecto orientado a la gestion digital de contenidos de Dungeons & Dragons 5a edicion. El repositorio revisado se organiza como un monorepo con un frontend en Angular 21 dentro de APP/ y un backend en FastAPI dentro de API/, ambos apoyados en MongoDB. Sobre esta base se ha construido una aplicacion web capaz de consultar catalogo oficial, crear contenido homebrew y gestionar personajes de usuario.

El ecosistema de herramientas para juegos de rol de mesa suele resolver muy bien la consulta pasiva de reglas, pero no siempre integra en una unica solucion el catalogo, la creacion de variaciones personalizadas, el mantenimiento de fichas completas y la generacion de documentacion util para partida. En este contexto, Grimledger se plantea como una plataforma propia, con control total sobre el modelo de datos y con capacidad de evolucionar hacia servicios mas avanzados.

La revision del codigo muestra ademas dos lineas de crecimiento ya previstas. La primera es un asistente de IA para formularios, documentado en PLAN_ASISTENTE_IA_FORMULARIOS.md, pensado para ayudar a crear personajes, monstruos, hechizos, backgrounds e items sin guardar automaticamente en backend. La segunda es un sistema de impresion en PDF para seleccionar varios elementos del mismo tipo, por ejemplo monstruos, items o personajes, y descargar dossiers detallados preparados para consulta o impresion.

1.2 Problematica actual

La preparacion de contenido para una campana de D&D exige combinar informacion oficial, adaptaciones caseras y documentos auxiliares. En la practica, esto suele obligar al usuario a saltar entre distintas fuentes para consultar razas, clases, hechizos, monstruos y equipo, mientras mantiene en otro lugar sus personajes o sus modificaciones homebrew. El resultado habitual es fragmentacion de informacion y sobrecoste de mantenimiento.

Tambien existe una segunda dificultad: la creacion y refinado de contenido complejo. Un monstruo con acciones, resistencias, spellcasting y CR razonable, o un personaje con inventario, rasgos, hechizos y bio bien organizados, requiere muchos campos y bastante consistencia interna. Por eso tiene sentido plantear una ayuda futura basada en IA, pero con una condicion fuerte: el modelo no debe tomar control de la persistencia ni operar libremente sobre la interfaz.

Por ultimo, una vez que el contenido ya existe, aparece la necesidad de distribuirlo y usarlo fuera de la aplicacion. Aqui entra la futura exportacion PDF: no se trata solo de imprimir una ficha suelta, sino de permitir seleccionar varios elementos homogeenos y generar un documento detallado y consistente, util para mesa, preparacion de sesion o archivo.

1.3 Solucion propuesta

La solucion propuesta por Grimledger se apoya en una arquitectura web desacoplada. El frontend Angular ofrece una SPA con rutas lazy, componentes standalone, listados con filtros y formularios ricos. El backend FastAPI concentra el acceso a MongoDB, la validacion de modelos y la carga de imagenes. La combinacion actual permite resolver varias necesidades reales del dominio:

  • Consulta del catalogo de razas, clases, backgrounds, hechizos, monstruos e items.
  • Creacion, edicion y borrado de contenido homebrew en backgrounds, hechizos, monstruos e items.
  • Creacion de personajes mediante asistente por pasos y gestion detallada de ficha.
  • Uso de una API propia sobre datos oficiales sembrados en MongoDB, evitando depender del tercero en tiempo de uso normal.

Sobre esta base implementada se plantean dos ampliaciones futuras coherentes con el sistema actual. La primera es un copiloto de formularios basado en IA, ejecutado con comandos tipados y proxy backend. La segunda es una capa documental de exportacion PDF por lotes homogeneos. Ambas lineas son compatibles con el codigo existente porque reutilizan la separacion actual entre estado local del frontend y servicios centralizados del backend.

2. Objetivos

El objetivo general de este Trabajo de Fin de Grado es desarrollar una aplicacion web para la gestion centralizada de contenidos de Dungeons & Dragons 5a edicion, con soporte real para catalogo, contenido homebrew y personajes, y dejar definida una base tecnica solida para futuras capacidades de asistencia inteligente y generacion documental en PDF.

Para alcanzar este objetivo general, se concretan los siguientes objetivos especificos:

  1. Construir una arquitectura web desacoplada: separar claramente frontend, backend y persistencia mediante Angular 21, FastAPI y MongoDB.
  2. Centralizar el catalogo de juego: ofrecer consulta unificada de razas, clases, backgrounds, hechizos, monstruos e items desde una API propia.
  3. Soportar contenido editable: permitir crear, editar y eliminar entidades homebrew del dominio, manteniendo una diferenciacion entre catalogo oficial y contenido generado por usuarios.
  4. Gestionar personajes de forma completa: incluir un flujo guiado de creacion y una ficha posterior con inventario, hechizos, rasgos y biografia.
  5. Preparar la integracion futura de IA: definir una estrategia segura para asistir en formularios sin que el modelo persista cambios automaticamente.
  6. Preparar la exportacion documental futura: plantear un sistema capaz de seleccionar multiples entidades del mismo tipo y descargar PDFs detallados y consistentes.

4. Analisis

4.1 Actores

El sistema se analiza a partir de tres actores principales, extraidos del comportamiento real del frontend y del backend, y de las ampliaciones previstas.

  1. Usuario no autenticado: puede acceder a login y registro, pero no debe operar con el resto de la aplicacion funcional. Su objetivo principal es autenticarse.
  2. Usuario autenticado: consulta catalogo, crea y gestiona personajes, y puede crear o editar contenido homebrew segun permisos y autoria.
  3. Administrador: hereda el comportamiento del usuario autenticado y, en el estado actual del frontend, se considera con capacidad de edicion completa sobre el contenido mostrado. Esta figura existe ya en el modelo logico mediante isAdmin, aunque la autorizacion backend todavia necesita refuerzo.

De cara al futuro, estos mismos actores seguiran siendo validos para las dos ampliaciones principales. El usuario autenticado sera quien invoque el asistente de IA sobre formularios y quien seleccione conjuntos homogeneos para exportacion PDF. El administrador seguira teniendo un papel reforzado en moderacion y gestion del contenido comun.

4.4 Diagrama de clases

El modelo conceptual del sistema gira alrededor de varias entidades de dominio. User representa al usuario autenticado. Character es la entidad mas rica y agrupa inventario, hechizos, rasgos y biografia. Background, Spell, Monster e Item representan contenido editable y tambien bases para el contenido oficial importado.

En el caso de Item, el analisis debe reflejar su naturaleza polimorfica: una misma coleccion fisica soporta armas, armaduras, herramientas, monturas, equipo aventurero y objetos magicos. Ademas, Race, Subrace, DndClass, Subclass, Trait y Feature actuan como soporte del catalogo y sirven tanto para consulta como para enriquecer la creacion de personajes.
